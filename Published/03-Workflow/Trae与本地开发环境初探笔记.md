

## 一、初始知识点

> 网页端对话式 AI 已经无法满足进阶的工程需求，搭建好高配的底层工具才是迈向 AI-Native 开发的第一步。

在开始这次系统的梳理之前，我主要的 AI 使用方式还停留在传统的网页端对话，这种模式上下文容易丢失，且无法直接操作本地代码库，难以带来实质性的技术提升。目前我已经搭建好了基础的“硬件”环境：本地安装了 Trae 作为 AI 驱动的 IDE，并且通过 WSL2/Docker 环境部署了 AnythingLLM 结合 Ollama 的本地知识库。同时，在日常开发中，我主要以 Python 为主，并习惯使用 Miniconda 将环境（如存放在 `D:\Miniconda\envs`）与具体的项目文件进行物理隔离，以方便集中管理。

## 二、学习过程中遇到的问题 & 如何解决的

> 习惯了网页端的纯文本交互后，初次接入具备本地执行权限的 AI IDE，必然会经历一段从“环境失控”到“精准驾驭”的阵痛期。

**1. Trae 提问响应过慢与无效的全局扫描**

- **发现过程：** 在 Trae 的对话框里，我有时只是想问一个基础的 Python 命令行问题，结果发现它每次都会去全局扫描一遍我的项目文件，导致响应极其缓慢。
- **我的原始思考：** 我以为这是新工具的通病，甚至考虑是不是每次问新问题都必须去新建一个空白的无文件项目。
- **学到的原理 / 迭代后的认识：** Trae 默认会将当前项目作为 Context（上下文）挂载到聊天中，以实现 Codebase 级别的理解。如果不加以干预，它确实会带着整个工程去思考简单问题。
- **解决方案总结：** 当询问与当前代码无关的通用问题时，直接在对话框输入区域点叉号 `x`，删掉带有 `@Workspace` 或当前文件名的标签。脱离了本地文件束缚后，它的回复速度和网页端一样快；或者直接点击 `+` 新建一个纯净的对话窗口。

**2. SOLO 模式带来的环境破坏风险与模式区分**

- **发现过程：** Trae 有时会使用沙盒来预测，甚至会在我不知情的情况下修改我本地的内容和终端命令行，这极有可能导致我本就脆弱的 Python 环境彻底崩溃。

- **我的原始思考：** 为了防止它乱动本地文件，我考虑是不是必须套一层虚拟机（VM）来进行物理隔离。

- **学到的原理 / 迭代后的认识：** 这其实是因为混淆了 Trae 的两种核心运行机制。它分为 IDE（Chat）模式和 SOLO（Agent/Builder）模式。

  > - **IDE 模式：** 传统的 Copilot 逻辑，我是主导者。AI 给出的代码会在编辑器生成 Diff 视图（红绿对比），必须由我手动点击 `Accept` 才会修改文件，绝对安全。
  > - **SOLO 模式：** 全自动智能体逻辑，AI 是主导者。它拥有读写文件和执行终端命令的最高权限，用于端到端执行任务。

- **解决方案总结：** 现阶段停止滥用 SOLO 模式。日常开发严格切换回 IDE 模式，通过 `Ctrl+I` 进行行内修改。修改前利用本地 Git 进行 Commit 存档，一旦 AI 把代码改崩，直接在源代码管理面板 `Discard Changes` 一键撤回。

**3. Conda 虚拟环境路径的识别与精准控制**

- **发现过程：** 为了验证环境隔离，我在 Trae 的终端（PowerShell）里使用了 `conda activate vision`，但随后用 `where.exe pip` 检查时，发现路径依然指向了 `D:\Miniconda\Mini\Scripts\pip.exe` 或全局路径，而不是我预期的 `D:\Miniconda\envs\vision`。
- **我的原始思考：** 我认为是由于我的项目文件夹和环境文件夹不在同一个地方导致的，或者 AI 在胡扯。
- **学到的原理 / 迭代后的认识：** 项目代码与依赖环境物理隔离是非常专业的做法。终端路径指代错误的核心原因在于 PowerShell 的安全策略拦截了 Conda 的底层环境变量切换脚本，导致表面上激活了，实际上系统底层的“安装工”并没有切换。与其和 PowerShell 的底层玄学死磕，不如直接掌握最硬核的绝对路径调用法。
- **解决方案总结：** 彻底抛弃 `activate` 激活步骤，采用绝对路径精准制导安装。
  1. 指定环境安装库：`D:\Miniconda\envs\vision\Scripts\pip.exe install requests`
  2. 利用 Conda 自带参数跨空操作：`conda install -n vision requests`

**4. 破除对“技能 (Skills)”的误解：利用 Markdown 约束 AI 思维**

- **发现过程：** 我看到界面的智能体配置里有“技能”等选项卡，以为需要编写多个复杂的 `.md` 文档和 `.py` 脚本才能实现高级功能，由于目前不打算接入生成图片等复杂外挂，便觉得无从下手。
- **我的原始思考：** 觉得既然没有复杂的脚本，Trae 的智能体可能无法实现我想要的那种高度定制化的回答。
- **学到的原理 / 迭代后的认识：** AI 编程软件中的 Skill（技能/规则）分为“软行为约束”和“硬能力外挂（MCP）”。绝大多数时候，我们需要的仅仅是前者。大模型本身已经具备了写出顶级代码的逻辑，所谓的 Skill 本质上是一段**结构化的 Prompt 设定**，是用来约束 AI“在这个项目中应该用什么姿势写代码”的员工手册。
- **解决方案总结：** 在项目根目录创建 `.traerules` 文件，或者在新建智能体的提示词框内，写入结构化的 Markdown 文本（包含角色设定、思考链路约束、回答输出格式等）。通过纯文本的格式强制约束，将 AI 从一个“话痨助理”转变为一个“严格按规范输出代码的技术总监”。

**5. 默认模型切换与本地 Ollama 模型的接入**

- **发现过程：** 在对话框右下角我只看到了 Auto 以及有限的三个默认模型选项，一开始不知道该如何选择，也不知道怎么把我费劲部署好的本地 Ollama 结合进去。
- **我的原始思考：** 我以为只能被迫使用系统默认的模型，或者本地模型只能在独立的软件里孤立使用。
- **学到的原理 / 迭代后的认识：** Trae 其实内置了目前最顶级的模型。应对不同场景要有“换脑”思维：写代码（尤其是前端）Claude 3.5 Sonnet 是王者；解复杂算法或深层 Bug，DeepSeek R1 更强。更重要的是，Trae 支持通过 API 桥接外部模型，让前端界面和本地算力结合。
- **解决方案总结：**
  1. **日常切换：** 养成在提问前根据任务复杂度点击右下角 `Auto ∨` 切换对应模型的习惯。
  2. **接入本地：** 在设置 -> Models -> Add Custom Model 中，通过配置类似于 `http://localhost:11434/v1` 的 API 路径，将我本地用 Ollama 跑起来的大模型也加入到下拉菜单的“自定义模型”分组中，实现私有化调用。

**6. 单机对话受限与多智能体（Multi-Agent）协同网络**

- **发现过程：** 我发现目前 Trae 里的直接对话，没有我和网页版 AI 聊得顺畅、聪明。
- **我的原始思考：** 我以为是 Trae 这个软件本身的对话能力不行，还没有找到合适的设置项。
- **学到的原理 / 迭代后的认识：** 网页版 AI 是一个“泛用型全才”，而高级的 AI IDE 更像是一个“公司”。在这个公司里，不能指望一个 AI 解决所有问题，而是需要建立“专业技术团队”。多智能体系统（Multi-Agent System）才是终极形态。
- **解决方案总结：** 在创建智能体界面，按职责拆分建立多个单一功能的 Agent（例如“Python 架构师”、“UI 切图仔”），并为它们编写专门的 Skill 约束。最关键的是，打开 **【可被其他智能体调用 (SOLO Only)】** 开关。这样在全局派发复杂任务时，主 Agent 就会自动扮演项目经理，把上一个子 Agent 的输出作为下一个的输入，实现流水线式的自动化开发。

## 三、最终产出

> 本次探索彻底理清了 AI 辅助编程工具与本地开发环境的协作边界，输出了以下几项关键的可落地技能。

- **一套绝对路径环境管理命令模板：** 掌握了绕过终端环境异常，直接调用 `D:\Miniconda\envs\vision\Scripts\pip.exe` 进行 `install`、`list`、`uninstall` 和导出 `requirements.txt` 的硬核方法。
- **AI IDE 模式防线建立：** 明确了 IDE (Chat) 模式的 Diff 审批机制结合本地 Git 存档，是防止代码库被 AI 意外污染的终极防线。
- **上下文(Context)清理技巧：** 学会了通过删除输入框标签 `@Workspace` 来中断 AI 的全局扫描，以此恢复单点提问的响应速度。
- **AI 思维约束框架初建：** 认知到通过结构化 Markdown（如 `.traerules`）来定义 Persona（角色）和 Workflow（工作流）是当前最具性价比的 Skill 配置方式。

## 五、个人成长小结

> **全文概括：从“对话框提问者”到“AI 研发管线调度者”的认知跃迁**

本次针对 Trae 的深度探索，对我而言是一次认知维度的重要破冰。我最大的收获在于跳出了“差生文具多”的内耗陷阱，意识到工具的核心能力（上下文理解、行内生成、Agent 模式）是通用的，把一款第一梯队的工具吃透，远胜于频繁更换软件。

我从单纯期待 AI 帮我“解答一个错误”，逐渐转变为思考如何将大模型作为核心组件接入我的开发流水线。我也真正理解了，向 AI-Native 进阶的路上，我不再是一个底层码农，而更像是一个技术 Manager：我需要通过 Git 控制风险，通过 `.traerules` 制定规范，通过绝对路径精准分配资源。接下来，我将依托目前建立的开发防线，逐步将本地闲置的 AnythingLLM 知识库利用起来，将沉淀的 Markdown 笔记喂入向量库，尝试实现本地脑库与 Trae 的双剑合璧。

**⚠️ 避坑指南：**

1. 千万不要在未经 Git 提交干净的分支上，贸然开启 SOLO 模式让 AI 大幅重构代码。
2. 在 PowerShell 终端下，不要过度迷信命令提示符前缀的 `(环境名)`，当行为诡异时，优先使用 `where.exe pip` 进行验身，或直接上绝对路径。
