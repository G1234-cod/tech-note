

# Docker 环境搭建

## 一、初始知识点

> 本次学习前，我已经在 Windows 11 上安装了 VMware Workstation Pro，并在其中运行着一个 Ubuntu 虚拟机。我对“虚拟机”有大致的理解，知道它是一个用软件模拟出来的独立计算机系统。同时，我模糊地知道 WSL 是“在 Windows 里运行 Linux”的某种方式，但对它的本质、与虚拟机的区别完全不清楚。我最终的目标是想在本地搭建一个能跑 AI 大模型的知识库系统。

- **虚拟机的基本认知**：我知道虚拟机（如 VMware）可以在 Windows 上运行另一个操作系统（如 Ubuntu），但每次都需要启动、登录，比较麻烦，且会固定占用分配给它的内存和 CPU 资源。我的电脑性能本身不算很强，所以一直担心额外开虚拟机会导致系统卡顿。

- **对“知识库”的初步概念**：我理解的“本地知识库”是一个可以导入自己的文档，然后通过对话形式提问，由 AI 大模型基于文档内容来回答的系统。我希望一切数据都留在本地，保护隐私。

- **已有的技术基础**：我熟悉 Windows 的文件管理、PowerShell 的基本操作（如切换目录、运行命令），也对命令行不排斥，愿意按照教程执行命令。但在 Linux 系统、Docker 容器等方面完全零基础。

## 二、学习过程中遇到的问题 & 如何解决的

> 整个学习过程就是不断打破错误认知、建立正确模型的过程。从最初混淆 WSL 与 VMware，到理解 Docker 的安装逻辑，再到踩中镜像加速与汉化文件替换的坑，最后还多了一个“文件管理器里的 Linux 文件夹”这个意外发现。每个问题背后都藏着更底层的原理。下面按认知顺序整理。

### 问题 1：我已经有 VMware 里的 Ubuntu，为什么教程还让我装 WSL 和 Ubuntu？这不是重复了吗？

- **发现过程**： 
  在搜索“本地知识库部署”教程时，几乎所有资料的第一步都是“安装 WSL2 并配置 Ubuntu”。我立刻产生困惑：我电脑上明明已经有 Ubuntu 虚拟机了，为什么还要在 Windows 里再装一个 Ubuntu？它们是不是同一个东西？如果直接在 Windows 上再装一遍，会不会和已有的虚拟机冲突，导致系统崩溃？

- **我的原始思考**： 
  我当时以为 Ubuntu 是一个像 QQ、浏览器一样的“软件”，可以直接在 Windows 上运行。我看到本地文件夹里也有 Ubuntu 相关的文件，猜想是不是能直接双击打开它。我的认知是：只要有个地方能运行 Linux 命令行就行，虚拟机虽然笨重，但至少已经有了，重新装一个似乎很浪费磁盘空间，也担心两个“虚拟机”会打架。

- **学到的原理 / 迭代后的认识**： 
  经过解释，我弄清了几个核心概念：
  1.  **Ubuntu 不是一个应用软件，而是一个完整的操作系统**。它不能像 `.exe` 文件那样直接双击运行，必须有一个“宿主环境”来承载。我在本地文件夹中看到的 Ubuntu 相关文件，要么是 WSL 的虚拟磁盘文件（`.vhdx`），要么是虚拟机的镜像文件（`.vmdk`），都只是存储载体，不是可执行程序。
  2.  **WSL2 和 VMware 是两种不同的 Linux 运行方式**： 
      - WSL2 是 Windows 内置的一个轻量级虚拟机，将 Ubuntu 子系统深度集成到 Windows 中，启动极快，内存可以动态回收，不跑模型时几乎不占资源。 
      - VMware Workstation 是第三方全虚拟化软件，它模拟了一整台完整的 PC，启动慢，资源占用是预先分配且固定的。 
  3.  **WSL2 的 Ubuntu 和 VMware 的 Ubuntu 是两套完全独立的系统**，它们的文件分别存储在不同的位置，数据互不影响，可以同时存在。 
  4.  **关于冲突**：在 Windows 11 和新版 VMware Workstation Pro 的环境下，两者基于 Windows Hypervisor Platform (WHP) 技术可以和平共存，**通常不需要手动禁用其中一个**。只有在极少数旧版 VMware 或同时开启了 Hyper-V 等企业安全功能时才会冲突。

- **解决方案总结**： 
  我不需要卸载 VMware 或放弃已有的 Ubuntu 虚拟机，也**不需要在 Win11 上重复安装一个“一模一样的” Ubuntu**。部署本地知识库要求的是一个能运行 Docker 的 Linux 环境，我的 VMware Ubuntu 理论上也能满足条件。但考虑到我电脑性能不强，开虚拟机会更卡，**最终决定放弃虚拟机方案，直接采用更轻量的 WSL2 作为主战场**。这个选择既避免了重复劳动，也解决了性能担忧。

### 问题 2：WSL2 和 Docker Desktop 到底安装到哪里了？如何把所有文件都放到 D 盘？

- **发现过程**： 
  在开始安装时，我就明确知道自己 C 盘空间紧张，必须在操作过程中把所有与 WSL、Docker 相关的文件下载到 D 盘。但我不清楚这些组件的文件结构：哪些是程序本身，哪些是会产生海量数据的工作区？安装过程中默认的路径又在哪里？

- **我的原始思考**： 
  我原本以为只要在安装  Desktop 时把安装路径选在 D 盘，就万事大吉了。但后来发现，Docker 的核心“重量级”数据（镜像、容器、虚拟磁盘）实际上默认存放在 C 盘的 `%LOCALAPPDATA%\Docker` 目录下，即使安装程序在 D 盘，这些运行时数据还是会撑爆 C 盘。

- **学到的原理 / 迭代后的认识**： 
  1.  **Docker Desktop 的安装分为两部分**： 
      - **程序文件**：可以自定义安装路径，占用空间不大（通常几百 MB）。 
      - **数据目录**（Disk image location）：存放一个名为 `DockerDesktop.vhdx` 的虚拟磁盘文件，所有镜像、容器、数据卷等都会写入这个文件，它才是空间的占用大户。即使把程序装在 D 盘，这个数据文件默认仍在 C 盘，需要手动迁移。 
  2.  **WSL2 子系统本身也是一个巨大的虚拟磁盘文件**：它通常存放在 `C:\Users\<用户名>\AppData\Local\Packages` 下，但可以通过 `wsl --install -d <发行版> --location "D:\WSL"` 来指定安装位置，一步到位安装到 D 盘。

- **解决方案总结**： 
  - **WSL2 (Ubuntu)**：以管理员身份运行 PowerShell，使用 `wsl --install -d Ubuntu-24.04 --location "D:\WSL"` 将 Ubuntu 子系统直接装到 D 盘。  
  - **Docker Desktop**：虽然我多次尝试用命令行 `--installation-dir` 参数将程序装到 `D:\Docker`，但反复遇到 staging 临时文件夹冲突的错误。最终我接受了将程序装在默认路径（实际安装命令中的 `--installation-dir="D:\Docker"` 最终成功，程序文件在 D 盘），并在**首次启动后立即**通过设置界面的 `Resources → Advanced → Disk image location`，将数据目录迁移到 `D:\DockerData`。 
  这样保证了所有会增长的核心数据都在 D 盘，C 盘只有必要的小型配置文件。

### 问题 3：Docker Desktop 安装时反复报错“当文件已存在时，无法创建该文件”

- **发现过程**： 
  在执行自定义安装路径命令时，安装过程总会中断，报出 `System.IO.IOException: 当文件已存在时，无法创建该文件` 的错误。错误路径指向 `D:\Docker Desktop.staging` 这个临时文件夹，而且两次错误指向的具体文件不同（第一次是 `Docker Desktop Installer.exe`，第二次是 `7zr.exe`）。

- **学到的原理 / 迭代后的认识**： 
  1.  **Docker 安装程序的 staging 机制**：安装程序会先将所有文件解压到一个名为 `安装路径.staging` 的临时文件夹，再通过文件级移动（file-by-file move）一次性搬到正式目录。 
  2.  **残留导致冲突**：如果上一次安装被中断或失败，这个 `.staging` 临时文件夹不会被自动清理。当下一次安装再次尝试解压并移动时，发现目标位置已经有同名文件，就会抛出 IOException。即使重启电脑，由于文件夹本身还存在，问题依旧。 
  3.  **静默安装参数的作用**：通过加入 `--quiet` 和 `--accept-license` 参数，可以跳过 GUI 向导，减少因用户交互延迟或窗口未响应而导致的中断概率。

- **解决方案总结**： 
  - **彻底清理残留**：以管理员身份在 PowerShell 中逐条执行删除命令，清除 `D:\Docker Desktop`、`D:\Docker Desktop.staging`、`C:\ProgramData\DockerDesktop`、`%LOCALAPPDATA%\Docker`、`%APPDATA%\Docker` 以及临时解压目录 `%LOCALAPPDATA%\Temp\DockerDesktopInstallers` 等所有相关残留。 
  - **使用静默安装命令**：在清理后立即运行带 `--quiet --accept-license` 参数的安装命令，并在执行完后通过 `Get-Content` 检查安装日志，最终确认 `Installation succeeded`。 
  - **万一仍失败**：可以改用官方 `.msi` 安装包，通过 `msiexec /i` 命令进行安装，这种方式能完全绕过 staging 目录机制。不过我这次最终清理后成功了，未用到此备选方案。

### 问题 4：Docker 无法从 Docker Hub 拉取镜像（网络连接失败 / 403 Forbidden）

- **发现过程**： 
  在安装成功后，我运行 `docker run hello-world` 测试，结果报错： 
  第一次是 `dial tcp ... connectex: A connection attempt failed`，说明根本无法连接到 Docker Hub 的服务器。 
  配置了阿里云镜像加速器后，又变成 `403 Forbidden`。

- **学到的原理 / 迭代后的认识**： 
  1.  **Docker 默认从 Docker Hub 拉取镜像**，服务器在国外，国内直接访问几乎必然超时或连接中断。 
  2.  **镜像加速器是反向代理**：它本质上是一个在国内部署的 Docker Registry 代理服务器。配置加速器后，Docker 的请求会先发到加速器地址，由加速器代为从 Docker Hub 拉取并缓存。 
  3.  **阿里云加速器失效**：我注册了阿里云账号并完成了个人实名认证，获取了专属加速地址 `https://tq7jblif.mirror.aliyuncs.com`，但仍然返回 403。这是因为近期阿里云对个人用户的 Docker 镜像加速服务进行了政策调整，已不再向个人提供该功能。 
  4.  **JSON 配置的语法要求**：在手动编辑 `Docker Engine` 的 JSON 配置时，因为 `"experimental": false` 这行后面漏掉了一个逗号，导致配置解析失败，Docker Desktop 无法启动。这说明 JSON 严格要求每个键值对之间用逗号分隔，最后一个键值对后面不加逗号。

- **解决方案总结**： 
  - **替换可用的镜像加速器**：放弃阿里云，改用社区目前可用的免费镜像源（如 `https://docker.1ms.run` 和 `https://docker.xuanyuan.me`），在 Docker Desktop 的 `Settings → Docker Engine` 中正确配置 `registry-mirrors` 字段。 
  - **纠正 JSON 语法**：补全缺失的逗号，确保所有配置项格式正确。 
  - **最终验证**：再次运行 `docker run hello-world`，成功拉取镜像并输出 `Hello from Docker!`，标志着 Docker 环境完全就绪。

### 问题 5：Docker Desktop 的版本号混淆（CLI 29.4.3 vs Desktop 4.74.0）

- **发现过程**： 
  在下载汉化包时，我看到 Assets 列表中有多个文件，其中包含 `DockerDesktop-4.74.0-Windows-x86.exe` 字样。但我运行 `docker version` 得到的输出显示 `Version: 29.4.3`。我开始质疑：我的版本到底是 29.4.3 还是 4.74.0？汉化包应该匹配哪一个？

- **学到的原理 / 迭代后的认识**： 
  - **Docker 版本分两层**： 
    - **Docker CLI（命令行客户端）版本**：`docker version` 中 `Client` 部分的版本号，如 `29.4.3`。它只是一个给你敲命令用的工具，可以单独升级。 
    - **Docker Desktop（桌面程序）版本**：带有图形界面、系统托盘、后台引擎的管理软件，如 `4.74.0`。它内部捆绑了特定版本的 CLI 客户端。在这个例子里，Docker Desktop 4.74.0 内部集成的就是 CLI 29.4.3。 
  - **汉化包匹配的是 Docker Desktop 的版本**，因为替换的 `app.asar` 文件是桌面程序的界面资源，不是命令行工具的资源。

- **解决方案总结**： 
  我确认自己的 Docker Desktop 版本为 4.74.0，下载了对应的 `app-Windows-x86.zip` 汉化包。

### 问题 6：汉化替换文件后 Docker Desktop 打不开 / 双击无反应

- **发现过程**： 
  我按照最初的教程只替换了 `app.asar` 文件，重启 Docker Desktop 后双击桌面快捷方式毫无反应。检查任务管理器，发现程序进程没有启动。

- **我的原始思考**： 
  我猜测可能是操作时没有完全退出 Docker Desktop（它可能还在系统托盘后台运行），导致文件替换不完整或被占用。我也怀疑过是汉化包版本不对，但版本已确认一致。

- **学到的原理 / 迭代后的认识**： 
  - **正确退出 Docker Desktop**：点击界面右上角的 X 并不会真正退出程序，它只会最小化到系统托盘的鲸鱼图标。必须**右键点击鲸鱼图标，选择 “Quit Docker Desktop”** 才能彻底关闭所有进程。 
  - **4.74.0 版本的完整性校验**：从该版本开始，Docker Desktop 增加了文件完整性校验。如果只替换 `app.asar`，程序启动时会检测到校验和不匹配，直接拒绝启动。需要同时替换 **三个部分**： 
    1. `resources/app.asar` 
    2. `resources/app.asar.unpacked` 文件夹 
    3. `frontend/Docker Desktop.exe` 主程序文件 
  - **备份的重要性**：在替换任何原始文件之前，必须将原文件重命名备份（如 `app.asar.backup`），以便出问题时能快速回退。

- **解决方案总结**： 
  - 强制通过任务管理器结束所有 Docker 残留进程。 
  - 从解压的汉化包中找到对应的三个文件/文件夹，分别覆盖到安装目录下的正确位置。 
  - 重新启动 Docker Desktop，界面成功切换为中文。 
  - 如果仍有问题，可随时用备份的原版文件恢复。

### 问题 7：为什么文件资源管理器里突然多了一个“Linux”节点，里面还有 docker-desktop 和 Ubuntu-24.04？

- **发现过程**： 
  当我装好 WSL 和 Docker 后，无意间打开“此电脑”（文件资源管理器），发现左侧导航栏除了熟悉的 C 盘、D 盘之外，赫然多了一个名为 **Linux** 的图标。点进去之后，里面竟然有两个文件夹：**`docker-desktop`** 和 **`Ubuntu-24.04`**。我第一反应是：这是不是我操作失误导致的，会不会是病毒或者系统出错了？

- **我的原始思考**： 
  我原本以为 WSL 里的 Ubuntu 是隐藏在命令行背后的“黑盒”，它的文件应该和 Windows 完全隔离，只能通过终端里的 `ls` 等命令访问。这个图形化入口突然出现，让我怀疑自己是不是不小心共享了文件夹，或者 Docker 把什么东西泄露到了我的 Windows 里，感觉很不安。

- **学到的原理 / 迭代后的认识**： 
  1.  **这不是 bug，而是 WSL2 提供的官方功能**：这个“Linux”节点是 Windows 利用 Plan 9 网络文件协议，将 WSL2 所有发行版的根目录映射成网络位置。我可以直接像浏览本地文件一样操作 Linux 里的文件。 
  2.  **两个文件夹的含义**： 
      - **`Ubuntu-24.04`**：就是我亲手安装的那个 Ubuntu 子系统的完整文件系统。点进去就能看到 `/home/gyx`、`/etc`、`/usr` 等目录。 
      - **`docker-desktop`**：这是 Docker Desktop 自动创建的、用于运行 Docker 引擎和所有容器的一个极简 WSL 发行版。它是 Docker 的“后台工作间”，我们在 `docker run` 启动的容器本质上就活在这里面。 
  3.  **环境隔离的直观体现**：这个节点清晰地展示了 `Ubuntu-24.04` 和 `docker-desktop` 是**两个完全独立的 Linux 环境**。我在 `Ubuntu-24.04` 里可以乱装软件，不会弄脏 Docker 的后院，反之亦然。 
  4.  **便利与风险并存**：虽然可以通过拖拽方式方便地在 Windows 和 Linux 之间传文件，但如果直接用 Windows 的记事本或 Word 去编辑 Linux 下的配置文件（如 `.bashrc`），可能会破坏文件的 Linux 权限属性，导致程序出错。**稳妥的做法是：只把这里当成文件传输的中转站，代码和配置的修改还是回到 Ubuntu 终端里用 `nano` 或 `vim` 进行。**

- **解决方案总结**： 
  - 无需解决任何问题，这是一个正常且非常有用的功能。 
  - 我学会了利用它来轻松完成 **Windows ↔ Linux 的文件复制粘贴**，这比用 `scp` 命令或设置共享文件夹要简单得多。 
  - 我牢记了**不要在 Windows 里直接编辑 Linux 配置文件**这个避坑要点，以保证系统稳定。

## 三、最终产出

> 通过这次完整的学习与实操，我不仅成功配置了本地 Docker 开发环境，更重要的是建立了一套对 WSL、虚拟化、容器运行时等概念的清晰认知体系。

- **1. 厘清了三种在 Windows 上运行 Linux 的方式**： 
  我理解了 WSL2（轻量级子系统）、VMware/VirtualBox（完整虚拟机）、双系统三者的本质区别、资源消耗模式、适用场景，以及它们之间是否存在冲突。知道在 Windows 11 + 新版 VMware 下，WSL2 和虚拟机可以共存，不需要手动“二选一”。

- **2. 掌握了 Docker Desktop 的安装结构与文件管理**： 
  我清楚 Docker Desktop 分为**程序文件**（可自定义路径）和**数据目录**（`.vhdx` 虚拟磁盘，真正占用空间，必须手动迁移）。我能用命令行完成静默安装，并能通过日志文件（`install-log-admin.txt`）诊断安装失败原因，特别是清理 staging 临时文件夹来解决重复安装冲突。

- **3. 能够独立解决 Docker 的网络问题**： 
  我知道 Docker Hub 在国内无法直连的原因，掌握了“镜像加速器”的原理和配置方法。我实际经历了阿里云加速器失效到切换社区可用源的完整排错过程，并纠正了 JSON 语法错误。现在我知道如何编辑 `Docker Engine` 配置并重启生效。

- **4. 正确区分 Docker CLI 版本与 Docker Desktop 版本**： 
  我不会再将 `docker version` 输出的 `29.4.3` 误认为是桌面程序的版本。汉化包、安装包等均以 Docker Desktop 版本（如 `4.74.0`）为准。

- **5. 完成了 Docker Desktop 的汉化，理解了版本校验机制**： 
  我不仅知道如何下载匹配版本的汉化包，还掌握了 4.74.0 版本的三个文件替换策略。更重要的是学会了“彻底退出程序 → 备份原文件 → 替换 → 验证”的安全操作流程。

- **6. 所有核心文件（WSL 子系统、Docker 程序、Docker 数据）全部存放在 D 盘**： 
  - WSL 的 Ubuntu-24.04 位于 `D:\WSL`  
  - Docker Desktop 程序文件位于 `D:\Docker`  
  - Docker 运行时数据（镜像、容器）位于 `D:\DockerData`

- **7. 理解了文件资源管理器中“Linux”节点的本质与用途**： 
  我明白了它是 WSL2 文件系统的 Plan 9 映射，可以用于便捷的文件传输，但不应用来直接编辑 Linux 配置文件。同时，通过它直观地看到了 `docker-desktop` 和 `Ubuntu-24.04` 两个独立环境的隔离。

## 五、个人成长小结

> 这次最大的收获不是“装好了一个软件”，而是**建立了一套面对陌生复杂系统的拆解与排查方法论**。

- **从“模糊类比”到“精确模型”**： 
  以前我把 WSL 和虚拟机都笼统地理解为“在 Windows 里跑 Linux 的工具”，现在我清楚它们的技术路径完全不同，消耗资源的方式、与 Windows 的集成深度也截然不同。特别是当文件资源管理器里出现“Linux”节点时，我一开始感到困惑甚至不安，但最终理解了这是 WSL 深度集成的一部分，是我高效操作 Linux 文件的捷径。

- **从“抄命令”到“读日志”**： 
  在反复遇到 Docker 安装冲突的过程中，我学会了不再慌张地反复重试同一命令，而是去找到日志文件，逐行理解安装流程。当我看到 `Phase 1: Staging`、`Phase 3: Activation (file-by-file move)` 这样的输出时，突然理解了 staging 目录的由来，也明白了为什么必须清理它。这种**把错误信息当成线索而非障碍**的心态转变，对我的技术成长至关重要。

- **从“试错”到“可控回退”**： 
  汉化失败后，我意识到任何对系统文件的修改，都必须先备份原文件，确保有一条安全的回退路径。这不仅是操作习惯的养成，更是一种风险意识的提升——在生产环境中，这个原则同样适用。

- **避坑指南**： 
  - **不要用右上角的 X 关闭 Docker Desktop**，它只是最小化，后台进程仍在运行。替换文件前务必在系统托盘右键选择 “Quit Docker Desktop”。 
  - **阿里云的个人 Docker 镜像加速服务已基本失效**，不要死磕。多准备几个社区公共镜像源作为备选。 
  - **JSON 配置文件里，最后一个键值对后面不加逗号**，这是新手最容易犯的语法错误，会导致服务启动失败。 
  - **下载汉化包前，务必确认 Docker Desktop 版本号（不是 CLI 版本号）**，版本不一致会直接导致 GUI 无法启动。 
  - **清理 staging 临时文件夹是解决“文件已存在”错误的根本方法**，仅重启电脑不会自动清除它。 
  - **千万不要用 Windows 记事本直接编辑“Linux”节点里的配置文件**，那会破坏文件权限。传文件可以，改配置请回终端。

此刻，我的 Windows 11 上已有一个完全就绪的 Docker 容器运行环境，所有核心数据都在 D 盘，界面已汉化，并且我可以随时通过文件资源管理器与 Linux 环境交换文件。下一步，我将直接用 `docker run` 命令部署本地知识库应用（如 MaxKB），无需再为基础环境操心。
