## 一、初始知识点

> 本次学习开始前，我已有的认知基础虽然零散，但为理解云服务器部署的全流程提供了必要的铺垫。

在正式开始本次大作业之前，我已经掌握了一些相关的基础知识和技能：

- **计算机网络基础**：我了解 OSI 七层模型和 TCP/IP 协议栈的基本概念，知道 HTTP 协议默认使用 80 端口、HTTPS 使用 443 端口，也明白 IP 地址是网络中定位服务器的唯一标识。这些知识让我能够理解后续安全组配置中"放行端口"的本质——就是在传输层允许特定端口的流量通过防火墙。

- **Linux 操作系统基础**：我在课堂上学习过 Linux 的基本命令，知道 `ssh` 是用于远程登录的安全协议，了解 root 是系统的超级管理员账户。虽然平时练习多在本地虚拟机上进行，但原理是相通的。

- **域名系统（DNS）的浅层认知**：在此之前，我对 DNS 的理解停留在"把域名翻译成 IP 地址"这个层面，知道这是访问网站的必经步骤，但对解析过程的细节、顶级域名之间的差异、以及 DNS 服务对访问速度的影响并没有深入思考过。

- **云服务的基本概念**：我知道云服务器（ECS）是一种按需租用的虚拟服务器，可以自由选择配置和操作系统，不需要自己购买物理硬件。但对于安全组、弹性公网 IP 等云原生概念的实操经验几乎为零。

这些基础知识虽然不够系统，但它们像散落的拼图碎片，在接下来的学习过程中逐渐被串联起来，形成完整的知识体系。

## 二、学习过程中遇到的问题 & 如何解决的

> 这一阶段的核心收获并非来自命令行的敲击，而是来自对"域名"这一看似简单概念的深度重新理解。

### 问题 1：域名后缀对网站访问速度有没有影响？

**发现过程**

在购买域名时，我面对 `.com`、`.cn`、`.top`、`.xyz` 等各种后缀，价格从几块钱到几十块不等。我的第一反应是：便宜的后缀会不会访问慢？不同后缀的域名在速度上到底有没有差异？我带着这个问题开始查资料、请教导师。

**我的原始思考**

坦率地说，我最初是把域名和 CDN 加速的概念搞混了。我以为域名本身就自带加速功能，不同后缀可能对应不同的"加速线路"，就像快递公司有普通快递和加急快递的区别。我还担心买了便宜的 `.top` 域名，用户访问我的网站就会比 `.com` 慢一拍。

**学到的原理 / 迭代后的认识**

通过导师的讲解和自己的深入研究，我彻底厘清了域名在网站访问流程中的真实角色：

1.  **域名本身不产生加速效果**
    域名本质上是一个**索引符号**，它的作用仅仅是在用户访问网站的第一步——DNS 解析阶段——将人类可读的字符（如 `gyx-ai.cn`）翻译成机器可读的 IP 地址（如 `123.xxx.xxx.xxx`）。这个翻译过程通常在几十到几百毫秒内完成，一旦解析完成，后续的数据传输就完全由服务器性能、带宽和网络路由决定了，与域名再无关系。把域名比作通讯录里的联系人名字，真正决定你能多快找到联系人并打通电话的，是你的手机性能和信号，而不是联系人名字本身的长短。

2.  **"翻译器"的分布才是关键差异**
    不同后缀的域名，其权威 DNS 服务器的部署位置和质量确实不同，但这属于 DNS 解析性能的范畴，而非域名后缀本身在"加速"。具体来说：
    - **`.com` 和 `.net`**：由 VeriSign 公司运营，在全球有超过 200 个解析节点，无论用户在哪个国家访问，都能就近完成 DNS 解析，解析速度稳定在几十毫秒级别。
    - **`.cn`**：由中国互联网络信息中心（CNNIC）管理，解析节点主要部署在国内，国内访问速度非常快，且在国内访问 `.cn` 域名时不会受到国际链路波动的影响。但值得注意的是，`.cn` 域名需要实名认证和 ICP 备案才能解析，这是政策的硬性要求。
    - **`.top`、`.xyz` 等新通用顶级域名**：这些域名由不同的注册局运营，解析节点数量和质量参差不齐。有些注册局为了节省成本，在全球可能只有寥寥几个解析节点，导致某些地区的用户在首次访问时需要跨洲际网络进行 DNS 查询，解析延迟可达几百毫秒甚至更高。

3.  **"翻译器"是可以更换的——DNS 加速服务的价值**
    这就引出了一个重要的认知跃迁：域名的权威 DNS 服务器虽然在注册时由注册局默认指定，但**我们完全可以把域名的 DNS 解析服务迁移到专业的 DNS 服务商**，如 Cloudflare、阿里云 DNS、DNSPod 等。这些服务商在国内外都有大量解析节点，并且通常免费提供基础的 DNS 加速服务。迁移后，用户请求域名解析时，会由这些专业的、就近的节点来响应，而不是去找域名注册局那些可能较远或较少的原始服务器。对于我的大作业来说，把 `gyx-ai.cn` 的 DNS 服务器改为阿里云或 DNSPod 的免费服务，比纠结选 `.com` 还是 `.cn` 更能实际提升访问体验。

**解决方案总结**

鉴于我的网站主要面向国内用户，且考虑到大作业的稳定性和合规性，我最终选择了 `gyx-ai.cn` 域名。`.cn` 后缀在国内的解析稳定性有 CNNIC 背书，实名认证和备案虽然多了一步，但对于部署在国内服务器上的网站来说本来也要走这个流程，并不增加额外负担。同时，我计划在域名解析环节使用阿里云提供的免费 DNS 服务，进一步优化解析速度。

## 三、最终产出

> 本阶段虽然没有敲一行代码，但建立了关于域名和 DNS 的正确心智模型，这是后续所有网络配置的理论基石。

- **域名的本质认知**：域名是互联网上的"门牌号索引"，其核心职责是在 DNS 解析阶段完成域名到 IP 的翻译，翻译完成后即退出通信流程。域名本身不参与数据传输，因此没有"加速"或"减速"的物理能力。

- **影响 DNS 解析速度的真正因素**：
  1.  **权威 DNS 服务器的部署范围**：节点越多，用户就近解析越快。`.com` 和 `.cn` 在这方面有保障，小众后缀则需谨慎。
  2.  **递归 DNS 服务器的质量**：用户使用的本地 DNS（通常是运营商或公共 DNS 如 114.114.114.114）的性能和缓存策略直接影响解析延迟。
  3.  **DNS 服务商的选择**：可将域名的 DNS 解析服务迁移至专业服务商（如 DNSPod、Cloudflare）来获得免费的节点加速和更稳定的解析服务。

- **域名注册的实操能力**：
  - 能够在云厂商控制台完成域名的搜索、选择和购买流程。
  - 理解不同后缀域名在价格、实名认证、备案要求和解析稳定性上的差异。
  - 能够根据目标用户群体（国内 or 海外）选择合适的域名后缀和注册商。

- **云服务器部署的前置认知**：
  - 明确了"购买域名"和"配置云服务器"是两条独立但需协同的主线，后续需要通过 DNS 解析（添加 A 记录）将它们关联起来。
  - 理解了安全组作为云服务器"虚拟防火墙"的作用，以及 22、80、443 端口各自的用途。

## 四、面试高频考点

> 本阶段涉及的面试题集中在网络基础中的 DNS 协议，这是初级运维和后台开发岗位的必考内容。

| 面试考点                                  | 对应知识点                                                   |
| :---------------------------------------- | :----------------------------------------------------------- |
| **从输入 URL 到页面展示，发生了什么？**   | DNS 解析在整个流程中的位置：浏览器缓存 → 系统缓存 → 递归查询 → 根域名服务器 → 顶级域名服务器 → 权威 DNS 服务器。本阶段覆盖了域名到 IP 的转换逻辑。 |
| **DNS 使用什么协议？默认端口是多少？**    | DNS 主要使用 UDP 协议，默认端口号为 53。对于超过 512 字节的响应或在区域传输时会使用 TCP。 |
| **什么是 DNS 缓存？它如何影响访问速度？** | 浏览器、操作系统、递归 DNS 服务器都会缓存解析结果。缓存命中时无需发起完整查询链，解析时间可降至 1ms 以内。TTL（生存时间）控制缓存时长。 |
| **A 记录和 CNAME 记录的区别是什么？**     | A 记录将域名直接指向 IPv4 地址；CNAME 将域名指向另一个域名（别名），常用于 CDN 加速或指向第三方服务。本阶段讨论了 A 记录的场景。 |
| **如何优化 DNS 解析速度？**               | 1. 使用节点分布广的 DNS 服务商；2. 合理设置 TTL 值（稳定期设长，变更期设短）；3. 使用 DNS Prefetch（`<link rel="dns-prefetch">`）提前解析第三方域名。 |

## 五、个人成长小结

> 从"域名就是个网址"到"域名是 DNS 解析链条的起点"，这个认知跃迁让我开始用工程师的视角去拆解用户操作的每一个微观步骤。

**我掌握了什么**

这段时间的学习让我建立了一个清晰的认知：一个看似简单的"打开网页"动作，背后是一系列精密的分工协作。域名、DNS、服务器、HTTP 协议各司其职，任何一个环节的延迟或故障都会影响最终的用户体验。我不再笼统地认为"网站慢"，而是能够初步定位——是 DNS 解析慢？是服务器处理慢？还是网络传输慢？

具体来说，我掌握了域名注册的实操流程，理解了 DNS 解析的层级结构，厘清了域名后缀、DNS 服务商和网站访问速度之间的真实关系，为后续配置 DNS 解析和 SSL 证书打下了理论基础。

**认知上的跃迁**

最大的跃迁在于：**我意识到在技术选型时，"默认配置"和"最优配置"之间存在巨大的优化空间。** 注册域名时默认使用注册局提供的 DNS 服务器是大多数人的选择，但懂得将 DNS 迁移到专业服务商，就能在几乎零成本的情况下获得显著的解析性能提升。这种"不满足于跑通，追求跑好"的工程思维，是本次学习给我最重要的启发。

**避坑指南**

1.  **混淆域名和 CDN 的概念**：初学者容易把域名的"加速"和 CDN 的"加速"混为一谈。域名只负责解析，CDN 才负责把数据缓存到离用户近的节点。两者是完全不同的层面。
2.  **忽视小众后缀的 DNS 稳定性**：不要被极低的价格诱惑，`.top`、`.xyz` 等后缀的 DNS 解析可能因节点不足而导致首次访问延迟明显，尤其在全球范围内。
3.  **以为买了域名就能直接访问**：域名注册只是第一步，后续必须完成 DNS 解析（添加 A 记录指向服务器 IP）并确保服务器端口（80/443）已放行，网站才能真正被外界访问。
